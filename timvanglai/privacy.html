Privacy Policy

Tìm sân cầu lông vãng lai — Last updated: April 25, 2026

1. Who this covers

This Policy describes how the Tìm sân cầu lông vãng lai web application and its backend API (the “Service”) collect, use, store, and share information when you use features that are implemented in the current product (including account access, court and listing features, groups, uploads, and payments).

2. Information we collect (mapped to the codebase)

2.1 Account & authentication

  • Email address, full name (display name), and Google subject identifier (sub) when you sign in with Google — obtained from the Google-issued ID token verified on the server.
  • Email address and a password (stored only as a one-way password hash on the server) when you use email/password login for an account that was provisioned that way.
  • JWT access token and refresh token: issued by the Service; refresh token fingerprints are stored in the database; tokens may be held in browser storage on your device for session continuity.
  • Role (for example, customer vs. administrator) and account status flags maintained in the user record.

2.2 Profile & in-app economy

  • Phone number, biography, avatar image (storage key or URL), Long Xu balance, VIP tier, VIP expiry, and optional custom VIP label (where applicable).
  • Badminton CV fields you edit (headline, objectives, achievements, contact summary, styling choices, optional avatar override).

2.3 Content you create

  • Courts: names, addresses (legacy and current administrative subdivisions as structured in the app), Google Maps URLs, optional owner phone, coordinates, photos, video links, cached public business metadata from Google (e.g., rating counts, place types), and optional reviewer notes.
  • Listings: titles, descriptions, dates, session schedules, skill labels, pricing, player counts, contact channels (e.g., Zalo/Facebook URLs, phone), payment QR images, and links to originating groups or schedules when used.
  • Groups (“vanguard”): names, descriptions, weekly slot templates, media, payment QR images, join links.
  • Pairing seeks: display name, gender, age range, handedness, area fields, narrative text, media, contacts.
  • Reports or admin actions tied to moderation workflows present in the product.

2.4 Payments

  • Records required to operate Pay2S pending payments and webhooks: transfer references, amounts, package identifiers, VIP selections, and payment status as modeled in the database (exact fields depend on processor payloads).

2.5 Technical & operational data

  • Server logs may be created by hosting infrastructure (standard HTTP logs, error logs) — scope depends on deployment, not fully specified in application source.
  • Vercel Analytics may run on the production web build and collect usage metrics according to Vercel’s documentation.
  • API traffic is subject to rate limiting parameters configured for the Service.

3. How we collect it

  • Direct input through forms in the web client.
  • Google OAuth (Sign-In) via Google Identity Services; the browser receives an ID token that the API verifies server-side.
  • Google Maps Platform APIs (server-side, using an API key) when resolving a Google Maps URL into place details that staff or tools import into court records — this processes public place information, not the contents of your personal Google account.
  • File uploads (images/videos) through authenticated upload endpoints; files are stored in cloud object storage and referenced by storage keys in the database.
  • Payment provider callbacks (Pay2S webhook) for confirming purchases.

4. How we use information (feature mapping)

  • Authenticate you and keep your session secure (JWT issuance, refresh rotation logic).
  • Display and search courts, listings, groups, and pairing posts you or others publish.
  • Operate Long Xu debits/credits and VIP entitlements after successful payment confirmation.
  • Store media you attach to profiles, courts, listings, groups, or QR images.
  • Maintain administrative workflows (court approval, bulk import helpers, reports).
  • Improve reliability and security (rate limiting, error monitoring as configured in deployment).

5. Google user data (Limited Use)

When you choose Sign in with Google, the Service verifies your Google ID token and reads only what is needed to operate accounts in this product:

  • Email address — unique account identifier and communication channel for account-related notices you request.
  • Name — prefills your display name in the profile.
  • Google account subject ID (sub) — links your Service account to your Google identity for future sign-ins.

We do not use Google user data to serve general advertisements. We do not sell your personal information. We do not access Gmail, Google Drive files, or Google Calendar through this sign-in flow based on the current implementation.

You can review and revoke the Service’s access to your Google account at any time via Google’s permissions page:
https://myaccount.google.com/permissions

6. Storage & retention

  • Primary application data is stored in a MongoDB database (including users, courts, listings, groups, payments-related documents, refresh token records, and content collections).
  • Binary uploads are stored in cloud object storage (Google Cloud Storage integration in code) and referenced from MongoDB.
  • Retention: the codebase does not define automatic deletion timelines. Data is kept for as long as your account exists and as needed to provide the Service, comply with law, resolve disputes, or enforce agreements. Deletion mechanics for end users are not fully specified in source — contact the development team using the link in the footer to request assistance.

7. Sharing

  • Other users see information you deliberately publish (for example, listing contact links, public court pages, group descriptions).
  • Google receives OAuth authentication requests when you sign in; Google Maps Platform receives server queries for public place data when URLs are resolved.
  • Pay2S (or successor payment integration configured in the deployment) receives payment instructions needed to complete top-ups or VIP purchases.
  • Hosting and infrastructure providers (for example, Vercel for the frontend in production, database host, object storage) process data as subprocessors necessary to run the Service.
  • We do not sell personal information.

8. Security

  • Passwords are stored using one-way hashing (bcrypt parameters defined in application constants).
  • Google ID tokens are verified with Google’s libraries against the configured OAuth client ID.
  • Authenticated API routes require valid JWTs; refresh tokens are stored hashed in the database.
  • Payment webhooks require a shared secret configured server-side.
  • No method of electronic storage is perfectly secure; we apply reasonable safeguards appropriate to a web service of this type.

9. Your rights & choices

  • You may review or update much of your profile data inside the app where those screens are available.
  • You may disconnect Google from the Service’s perspective by revoking access in your Google account settings (you may lose Google-based sign-in until you reconnect).
  • Depending on applicable law, you may have rights to access, correct, delete, restrict, or port data. Because workflows vary by deployment, submit requests through the contact page linked in the footer.

10. Children’s privacy

The Service is intended for a general audience of sports participants and is not directed to children under 13 (or the minimum age required locally). Pairing and profile features may collect age ranges you supply. If you believe a child has provided personal information without appropriate consent, contact us via the footer link.

11. International transfers

Servers and subprocessors may be located outside your country of residence. By using the Service, you acknowledge that information may be processed where providers operate facilities.

12. Changes

We may update this Policy to reflect product or legal changes. The “Last updated” date will be revised; material changes may be communicated through the Service or contact channels where practicable.

13. Contact

For privacy inquiries, use the development team contact page (also repeated in the site footer):
https://laughingthi00.github.io/app-policy/contact/

1. Phạm vi

Chính sách này mô tả cách ứng dụng web Tìm sân cầu lông vãng lai và API phía máy chủ (“Dịch vụ”) thu thập, dùng, lưu và chia sẻ thông tin khi bạn dùng các tính năng đã triển khai (tài khoản, sân, tin tìm người, nhóm, tải lên, thanh toán).

2. Thông tin thu thập (theo code)

2.1 Tài khoản & xác thực

  • Email, họ tên hiển thị, mã định danh Google (sub) khi đăng nhập Google — lấy từ ID token do Google cấp, xác minh trên server.
  • Email và mật khẩu (chỉ lưu dạng băm một chiều) nếu dùng đăng nhập email/mật khẩu.
  • JWT và refresh token: refresh được lưu dạng băm trong DB; token có thể nằm trong bộ nhớ trình duyệt.
  • Vai trò (khách/quản trị) và cờ trạng thái tài khoản.

2.2 Hồ sơ & kinh tế trong app

  • SĐT, tiểu sử, ảnh đại diện, số dư Lông xu, tier VIP, hạn VIP, nhãn VIP tùy chỉnh (nếu có).
  • Dữ liệu CV cầu lông bạn nhập.

2.3 Nội dung bạn tạo

  • Sân: tên, địa chỉ, URL Maps, tọa độ, ảnh/video, metadata công khai từ Google (ví dụ điểm đánh giá), SĐT chủ sân tùy chọn.
  • Listing: tiêu đề, mô tả, ngày, ca, mức giá, số người, kênh liên hệ, ảnh QR, liên kết nhóm/lịch nếu dùng.
  • Nhóm vanguard: mô tả, lịch tuần, media, QR, link tham gia.
  • Tìm ghép: tên hiển thị, giới tính, độ tuổi, tay thuận, khu vực, mô tả, media, liên hệ.
  • Báo cáo/quy trình quản trị có trong sản phẩm.

2.4 Thanh toán

  • Bản ghi phục vụ Pay2S (giao dịch chờ, webhook): mã giao dịch, gói, VIP, trạng thái theo mô hình DB.

2.5 Kỹ thuật

  • Nhật ký máy chủ tùy môi trường triển khai.
  • Vercel Analytics có thể chạy trên bản production.
  • Giới hạn tần suất API theo cấu hình.

3. Phương thức thu thập

  • Nhập trực tiếp trên web.
  • Google OAuth; server xác minh ID token.
  • Google Maps Platform phía server khi resolve URL Maps thành thông tin địa điểm công khai — không truy cập Gmail/Drive/Calendar của bạn qua luồng đăng nhập hiện tại.
  • Upload file qua API có xác thực; lưu object storage.
  • Webhook Pay2S.

4. Mục đích sử dụng

  • Xác thực và duy trì phiên.
  • Hiển thị/tìm kiếm sân, listing, nhóm, tin ghép.
  • Vận hành Lông xu và VIP sau thanh toán.
  • Lưu media bạn đính kèm.
  • Quy trình quản trị (duyệt sân, nhập hàng loạt, báo cáo).
  • An toàn và vận hành (rate limit, log lỗi theo triển khai).

5. Dữ liệu người dùng Google

Khi bạn Đăng nhập bằng Google, Dịch vụ xác minh ID token và chỉ dùng phần cần thiết:

  • Email — định danh tài khoản.
  • Tên — điền sẵn tên hiển thị.
  • sub — liên kết tài khoản với Google cho lần đăng nhập sau.

Chúng tôi không bán dữ liệu cá nhân. Không dùng dữ liệu Google để quảng cáo bên thứ ba theo phạm vi triển khai hiện tại. Không truy cập Gmail/Drive/Calendar qua luồng này.

Bạn có thể xem và thu hồi quyền tại:
https://myaccount.google.com/permissions

6. Lưu trữ & lưu giữ

  • Ứng dụng dùng MongoDB làm DB chính.
  • File trên object storage (tích hợp GCS trong code).
  • Code không quy định thời hạn xóa tự động; dữ liệu giữ trong phạm vi vận hành Dịch vụ và nghĩa vụ pháp lý. Yêu cầu xóa qua liên hệ chân trang.

7. Chia sẻ

  • Người dùng khác thấy phần bạn công khai (listing, trang sân, nhóm).
  • Google xử lý OAuth; Google Maps Platform nhận truy vấn địa điểm công khai khi resolve URL.
  • Pay2S nhận thông tin cần để thanh toán.
  • Nhà cung cấp hosting (ví dụ Vercel), DB, storage xử lý với tư cách phụ trợ.
  • Không bán dữ liệu.

8. Bảo mật

  • Mật khẩu băm bcrypt.
  • ID token Google xác minh bằng thư viện Google và client ID.
  • API có bảo vệ JWT; refresh token lưu băm.
  • Webhook thanh toán yêu cầu bí mật cấu hình.

9. Quyền & lựa chọn

  • Xem/sửa nhiều mục trong app khi có màn hình tương ứng.
  • Thu hồi Google tại tài khoản Google.
  • Quyền theo luật địa phương (truy cập, sửa, xóa…) — gửi yêu cầu qua trang liên hệ chân trang.
